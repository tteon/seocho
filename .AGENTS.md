# .AGENTS.md

This file supplements the root `AGENTS.md`. The root file remains the canonical
execution contract for all coding agents. This file defines SEOCHO's shorthand
for execution plans and the local decision style agents should use when a task
is too complex for a direct edit.

## ExecPlans

When writing complex features or significant refactors, use an ExecPlan from
design to implementation. An ExecPlan is a living, self-contained execution
document written according to `.PLANS.md`.

Use an ExecPlan for:

- semantic retrieval, public memory answering, evidence bundles, Graph-RAG, or
  indexing/query behavior changes
- ontology control-plane, route profile, answer-shape, provenance, or
  insufficiency-policy changes
- provider, tracing, benchmark, profiling, or evaluation changes whose result
  must be interpreted rather than merely compiled
- architecture, workflow, migration, or multi-module refactors
- work where the implementation depends on ambiguous product judgment

Do not require an ExecPlan for trivial typo fixes, tiny test-only updates,
mechanical formatting, or a narrow one-file bug fix with an obvious acceptance
test, unless the user asks for one.

## SEOCHO Execution Posture

`.beads` is the canonical planning and status tracker. An ExecPlan is the
implementation spec, not a replacement for Beads. Create or claim the `bd` item
first, then use Gastown only for shared-seam reservations that need cross-agent
coordination.

Preserve the stack baseline unless a specific ADR changes it:

- OpenAI Agents SDK
- vendor-neutral tracing contract, with Opik as the preferred team backend
- DozerDB as the production graph backend
- single-tenant MVP with `workspace_id` propagated through runtime contracts
- Owlready2 only in the offline ontology governance path

Prefer MARA for LLM-as-judge and experiment synthesis. Keep OpenAI API usage
minimal. For embeddings, prefer local BGE or sentence-transformers style
embeddings before OpenAI embeddings, and use OpenAI embeddings only when the
experiment explicitly requires them.

## Evidence Swarm Rule

SEOCHO's swarm is not a default "agents debate the answer" mode. Swarm means
that indexing and query paths can collect these signals in parallel:

- ontology signal: evidence that the active ontology/profile is helping,
  missing, stale, or mismatched
- required slot: a field the question must fill to answer correctly
- relation path: the graph edge pattern needed to connect the entities and
  slots
- provenance: the source record, chunk, edge, query row, or derivation that
  supports a claim
- insufficiency: a named missing or weak part of the evidence that should stay
  visible to the answerer

The output of swarm work is one typed evidence bundle. The bundle should be
diffable, traceable, and suitable for answer synthesis, evaluation, and
debugging. Debate remains a separate product mode and should be introduced only
when comparing independent answer proposals is the feature under test.

## Ambiguity Review Panel

When a decision is ambiguous, do not resolve it by taste alone. Use a written
three-lens review and record the result in the ExecPlan `Decision Log`:

- `professor_agent`: checks semantic validity, research fit, and whether the
  design reflects lessons from relevant classic and modern papers
- `software_engineer_agent`: checks API contracts, tests, maintainability,
  type boundaries, and failure-mode visibility
- `computer_systems_agent`: checks latency, cost, cache behavior, profiling,
  observability, and operational safety

These are review lenses, not automatic answer-debate agents. Keep the review
concrete: state the decision, the evidence, the tradeoff, and the validation
that would falsify the choice.

## Validation Bias

For retrieval and reasoning work, a passing unit test is not enough. Validate
with the smallest meaningful ladder:

1. focused deterministic tests for the changed contract
2. medium E2E run when the change affects indexing/query quality
3. MARA-based judging for answer quality or evidence-use judgments
4. profiling when the change adds orchestration, native code, provider calls,
   larger context, or parallel work

Report regressions honestly. If a prompt or feature improves one query but
hurts the aggregate score, keep the evidence and either narrow the feature or
discard it.

## Landing Discipline

Keep changes scoped and commit frequently at meaningful milestones. Do not
stage unrelated dirty files. Before pushing, run the relevant validation gates,
update the Beads item, run `bd bootstrap` best effort, and push to `main` unless
the user explicitly asks for a different target.
