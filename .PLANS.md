# Codex Execution Plans (ExecPlans)

This document describes the requirements for an execution plan, called an
ExecPlan. An ExecPlan is a living design-and-implementation document that a
coding agent or a human novice can follow to deliver a working SEOCHO feature
or system change.

An ExecPlan must be self-contained. Treat the reader as a complete beginner to
this repository. They have only the current working tree and the single plan
file. They do not remember previous discussions, prior plans, or private
context.

## How To Use ExecPlans

When authoring an ExecPlan, follow this file to the letter. If this file is not
in context, read it from the repository root before writing the plan. Also read
the required repository context from `AGENTS.md`: `README.md`, `CLAUDE.md`,
`docs/WORKFLOW.md`, `docs/ISSUE_TASK_SYSTEM.md`, and
`docs/decisions/DECISION_LOG.md`. If the work touches semantic retrieval,
public memory answering, indexing/query quality, evidence bundles, or Graph-RAG
behavior, also read `docs/GRAPH_RAG_AGENT_HANDOFF_SPEC.md`.

When implementing an ExecPlan, do not prompt the user for next steps. Proceed
to the next milestone, revise the plan as progress is made, and keep every
living section current. Resolve ambiguity autonomously when it can be resolved
from repository context, but record the decision and the reason in the plan.
Commit frequently at meaningful milestones.

When researching a challenging design, use milestones for proof-of-concept
work. A proof of concept is a small, testable implementation that answers one
feasibility question before the full design is promoted. Keep prototypes
additive, measurable, and easy to remove.

## Non-Negotiable Requirements

Every ExecPlan must remain fully self-contained. Include all knowledge and
instructions needed for a novice to succeed. Do not rely on earlier chat
messages, private notes, or external blogs.

Every ExecPlan is a living document. Contributors must revise it as progress is
made, discoveries occur, and design decisions are finalized. Each revision must
remain self-contained.

Every ExecPlan must produce demonstrably working behavior. Do not define
success as "code was added." Define what a human can run and observe.

Every ExecPlan must define terms of art in plain language. If a phrase such as
"ontology signal", "relation path", "daemon", "middleware", "Graph-RAG", or
"typed evidence bundle" appears, define it where it first appears and explain
where it manifests in this repository.

## Formatting

When sending an ExecPlan in chat, wrap the entire plan in one fenced code block
labeled `md`. Do not nest additional triple-backtick fences inside that block.
When writing an ExecPlan to a Markdown file where the file is only the plan,
omit the outer fence.

Use plain prose. Prefer sentences over tables and long lists. Checklists are
permitted only in the `Progress` section, where they are mandatory. Use two
blank lines after every heading. Use repository-relative file paths and exact
commands.

When commands, transcripts, diffs, or code snippets are needed inside an
ExecPlan, indent them instead of using nested fences.

## SEOCHO Operating Constraints

Every ExecPlan must preserve SEOCHO's current baseline unless the plan includes
an explicit ADR-backed reason to change it:

- OpenAI Agents SDK for agent orchestration.
- Vendor-neutral tracing as the contract, with Opik as the preferred team
  backend.
- DozerDB as the production graph backend.
- Single-tenant MVP behavior with `workspace_id` propagated through runtime and
  data contracts.
- Owlready2 only in the offline ontology governance path, never as hot-path
  reasoning.

Use `.beads` as the canonical planning and status tracker. Gastown is only a
coordination plane for shared-seam reservations. An ExecPlan may describe work,
but it does not replace `bd` status, sprint labels, or roadmap labels.

Use centralized configuration and do not hardcode secrets. Prefer logging over
print statements. Keep changes scoped, typed, testable, and free of broad
hidden side effects.

Prefer MARA for LLM-as-judge, qualitative experiment synthesis, and model
judgment loops. Keep OpenAI API usage minimal. Prefer local BGE or
sentence-transformers style embeddings when embeddings are needed, and use
OpenAI embeddings only when the validation question specifically requires them.

## SEOCHO Evidence Swarm Contract

For SEOCHO, swarm is an evidence assembly layer, not a default answer-debate
layer.

An ontology signal is a typed observation that the active ontology or semantic
profile helped, failed, drifted, or needs review. In this repository it appears
in ontology control-plane work, route profiles, semantic artifacts, and
evidence-bundle metadata.

A required slot is a field that must be filled to answer the question. For
example, a financial change query may need company, metric, start period, end
period, and operation.

A relation path is the graph pattern that connects entities and slots. It is
the answer to "which graph edges must exist for this answer to be grounded?"

Provenance is inspectable support for a claim: a source record, chunk, graph
edge, query row, benchmark case, or derivation receipt.

Insufficiency is a named missing or weak part of the evidence. SEOCHO should
preserve insufficiency through answer synthesis instead of hiding it behind a
fluent answer.

A typed evidence bundle is a structured payload that collects intent, selected
evidence, slot fills, missing slots, provenance, confidence, and derived-value
receipts. It should be diffable, traceable, and suitable for evaluation.

When an ExecPlan changes indexing or query behavior, it must describe how
ontology signals, required slots, relation paths, provenance, and insufficiency
are collected and assembled into a typed evidence bundle. Debate should be
introduced only when the feature is explicitly about comparing independent
answer proposals.

## SEOCHO Review Panel

When requirements are ambiguous, use a written review panel in the ExecPlan.
This panel is a reasoning discipline, not a requirement to spawn runtime agents.
Record the result in `Decision Log`.

The `professor_agent` lens checks research fit, semantic validity, and whether
the design reflects relevant classic and modern work. In SEOCHO, this lens
protects against adding graph-shaped complexity that does not improve grounded
answering.

The `software_engineer_agent` lens checks contracts, tests, maintainability,
typed interfaces, and failure-mode visibility. In SEOCHO, this lens protects
against features that pass a demo but cannot be debugged or extended.

The `computer_systems_agent` lens checks latency, cost, cache behavior,
profiling, observability, and operational safety. In SEOCHO, this lens protects
against orchestration that is elegant but too slow or expensive to ship.

The plan must state the decision, the tradeoff, the evidence used, and the
validation that would change the decision.

## Required Sections

Every ExecPlan must contain these sections and keep them current:

- `Purpose / Big Picture`
- `Progress`
- `Surprises & Discoveries`
- `Decision Log`
- `Outcomes & Retrospective`
- `Context and Orientation`
- `Plan of Work`
- `Concrete Steps`
- `Validation and Acceptance`
- `Idempotence and Recovery`
- `Artifacts and Notes`
- `Interfaces and Dependencies`

For semantic retrieval, evidence swarm, Graph-RAG, profiling, benchmark, or
provider work, also include:

- `SEOCHO Evidence Contract`
- `SEOCHO Review Panel`
- `Cost, Latency, and Provider Policy`

## Purpose Comes First

Begin by explaining why the work matters from a user's perspective. State what
someone can do after this change that they could not do before, and how they
can see it working.

If the change is internal, explain how its impact can still be demonstrated.
For example, state which test fails before the change and passes after, or
which E2E scenario now produces better evidence coverage without hiding missing
slots.

## Milestones

Milestones are narrative. Each milestone must explain the goal, what will exist
after it, the files likely to change, the commands to run, and the observable
acceptance result.

Each milestone must be independently verifiable and incrementally implement the
overall goal. If a milestone is a prototype, label it as prototyping and state
the criteria for promoting or discarding it.

## Validation And Acceptance

Validation is not optional. An ExecPlan must include exact commands to run from
the repository root and expected observations.

For normal code changes, include focused tests. For broader changes, include
the relevant CI or smoke command. Examples:

    uv run pytest seocho/tests -q
    bash scripts/ci/run_basic_ci.sh
    scripts/pm/lint-agent-docs.sh

For semantic retrieval or evidence-bundle changes, include a medium E2E or
benchmark run when feasible. Prefer MARA for LLM-as-judge. A representative
MARA-first command should make provider and model explicit, for example:

    uv run python scripts/benchmarks/run_finder_benchmark.py \
      --dataset /path/to/finder_sample.json \
      --mode local \
      --scenario advanced \
      --provider deepseek \
      --model deepseek/deepseek-chat \
      --limit-per-category 20

If the repository provides a profiling tool for the affected path, include it
when the change adds orchestration, provider calls, larger context, native code,
or parallel work. Treat profile data as a gate before activating optimization
work by default.

Report exact gaps. If the full suite is not run, state what was run, what was
not run, and why.

## Idempotence And Recovery

Steps must be safe to repeat when possible. If a command can fail halfway, say
how to retry. If a migration or destructive action is necessary, include a safe
backup or rollback path.

Do not tell future agents to delete, reinstall, or reset `.beads` as a first
response. Use the Beads recovery guidance in `AGENTS.md` and
`docs/ISSUE_TASK_SYSTEM.md`.

## Skeleton Of A Good ExecPlan

The following skeleton is the minimum shape. Expand it with concrete repository
paths, commands, and expected observations.

    # Short, action-oriented description

    This ExecPlan is a living document. The sections `Progress`,
    `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective`
    must be kept up to date as work proceeds.

    This plan follows `.PLANS.md` from the repository root.

    ## Purpose / Big Picture

    Explain what someone gains after this change and how they can see it
    working. State the user-visible behavior or measurable internal behavior
    that will exist after implementation.

    ## Progress

    - [x] (2026-06-04 12:00Z) Example completed step.
    - [ ] Example remaining step.
    - [ ] Example partially completed step, with completed and remaining work
          stated explicitly.

    ## Surprises & Discoveries

    - Observation: Describe an unexpected behavior, bug, optimizer result, or
      benchmark result.
      Evidence: Include the short command output, metric, trace field, or file
      path that proves the observation.

    ## Decision Log

    - Decision: State the decision.
      Rationale: State the tradeoff and why this choice fits SEOCHO.
      Date/Author: 2026-06-04 / Codex

    ## Outcomes & Retrospective

    Summarize outcomes, gaps, and lessons learned at major milestones and at
    completion. Compare the result against the original purpose.

    ## Context and Orientation

    Explain the current state as if the reader knows nothing. Name the relevant
    files and modules by repository-relative path. Define every non-obvious
    term before using it.

    ## SEOCHO Evidence Contract

    If the work touches indexing, query, Graph-RAG, or answer synthesis,
    describe the ontology signals, required slots, relation paths, provenance,
    insufficiency states, and typed evidence bundle fields that must exist at
    the end of the plan.

    ## SEOCHO Review Panel

    Record the professor_agent, software_engineer_agent, and
    computer_systems_agent review when ambiguity exists. State the final
    decision and what evidence would falsify it.

    ## Cost, Latency, and Provider Policy

    State whether the change uses MARA, OpenAI, local embeddings, BGE,
    sentence-transformers, DozerDB, Opik, or another provider/service. Explain
    why the choice is necessary and how cost or latency will be measured.

    ## Plan of Work

    Describe the sequence of edits. For each edit, name the file, function, or
    module and what to insert or change. Keep it concrete and minimal.

    ## Concrete Steps

    State exact commands to run from the repository root. Include concise
    expected output when it helps a novice distinguish success from failure.

        uv run pytest path/to/test_file.py -q

    ## Validation and Acceptance

    Phrase acceptance as behavior a human can verify. For example: running the
    focused test command reports all tests passed, the E2E artifact shows no
    zero-write indexing run, and the evidence bundle exposes missing slots
    instead of fabricating them.

    ## Idempotence and Recovery

    Explain which steps are safe to repeat, how to recover from partial
    failures, and how to keep the workspace clean.

    ## Artifacts and Notes

    Include short transcripts, diffs, metrics, or trace excerpts that prove
    success. Keep them concise and focused.

    ## Interfaces and Dependencies

    Name the libraries, modules, services, types, and function signatures that
    must exist at the end of the milestone. Be prescriptive enough that a novice
    can implement without guessing.

## Revision Notes

When revising an ExecPlan, update every affected section. At the bottom of the
plan, add a note describing what changed and why. A future agent must be able
to restart from the revised plan alone.
