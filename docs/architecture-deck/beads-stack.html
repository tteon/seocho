Problem

markdown TODO로는 에이전트 규모가 안 버텨진다

에이전트가 여러 세션·여러 브랜치에서 동시에 일할 때 일반 이슈 트래커가 깨지는 지점들:

컨텍스트 휘발

세션이 끝나면 에이전트 머릿속이 리셋. "이 이슈가 왜 만들어졌는지" 단서가 PR 설명 밖에 없음.

ID 충돌

시퀀셜 ID(#123)는 브랜치마다 다른 값이 배정됨 → 머지 시 충돌.

동시 쓰기 손실

여러 에이전트가 같은 이슈를 동시에 업데이트하면 "last write wins"로 필드가 지워짐.

필요한 건 세 축: 영속 메모리안전한 동시성물리적 격리. 각각 beads · Dolt · git worktree가 맡는다.

01

beads

AI 에이전트용 dependency-aware 이슈 그래프

github.com/gastownhall/beads · 21.1k stars

Concept

beads가 일반 트래커와 다른 네 가지

1. 의존성 그래프

이슈 사이에 blocks, blocked-by, discovered-from, relates-to, supersedes 관계가 1급 시민.

bd ready가 지금 풀린(=블로커가 없는) 일만 돌려준다 — 에이전트에게 "다음 뭐 할까"를 자동으로 알려주는 핵심.

2. 해시 기반 ID

시퀀셜 번호가 아니라 bd-a1b2 같은 해시. 두 브랜치가 동시에 이슈를 만들어도 ID 충돌 0.

머지가 깨끗해진다. 에픽은 bd-a3f8.1.1처럼 점 표기로 계층 확장.

3. 에이전트 전용 출력

기본적으로 --json. 모든 필드가 기계 파싱 친화적.

bd ready --json, bd show bd-42 --json → 스크립트/에이전트가 바로 사용.

4. 영속성 + 감사

백엔드가 Dolt(=버전 SQL). 모든 변경이 커밋으로 남음.

"이 라벨이 언제 왜 바뀌었지?" → bd dolt log로 되짚기.

Commands

에이전트 한 사이클의 bd 호출

# ① 준비된 일 탐지
bd ready --json

# ② 원자적 claim
bd update bd-42 --claim --json

# ③ 상세 & 히스토리
bd show bd-42 --json

# ④ 작업 중 파생 이슈를 부모에 연결
bd create "Found edge case in loader" \
  --description "..." -p 1 \
  --deps discovered-from:bd-42 --json

# ⑤ 의존성 추가
bd dep add bd-99 bd-42   # 99가 42를 블록

# ⑥ 완료
bd close bd-42 --reason "Done"

세 가지 포인트

  • --claim 한 명령으로 "내 것" + "in_progress" 두 상태를 원자적으로 전이. 두 에이전트가 동시에 --claim 해도 한 쪽만 성공.
  • discovered-from이 가장 저평가된 관계. 작업 중 발견한 새 이슈를 부모에 링크해두면 "왜 이 이슈가 생겼지?" 답이 그래프에 남는다.
  • --json은 에이전트가 제어문으로 반응할 수 있게 해주는 기본 플래그. jq 조합과 궁합이 좋다.
02

Dolt

git for data — 버전 관리되는 SQL 데이터베이스

Cell-Level Merge

같은 행을 두 명이 동시에 고쳐도 — 살아남는다

Dolt의 핵심 차별점. 머지가 셀(cell) 단위로 일어난다.

보통 DB / last-write-wins

T1 writes: bd-42.priority = 1
T2 writes: bd-42.status   = done
---
최종: priority=2(디폴트), status=done
     ← T1의 priority 업데이트 유실

같은 row를 둘이 건드리면 한 쪽이 덮어써진다.

Dolt / cell-level merge

T1 writes: bd-42.priority = 1  (branch A)
T2 writes: bd-42.status   = done (branch B)
---
merge A + B → priority=1, status=done
     ← 둘 다 살아남음

서로 다른 컬럼을 고쳤으면 자동 병합. 같은 컬럼일 때만 충돌 처리.

beads가 AI 에이전트 여러 명이 동시에 돌아가도 데이터를 안 잃는 이유가 여기에 있다. 이슈 레코드는 협업 문서처럼 부분 업데이트된다.
Branching & Modes

DB도 브랜치한다 — 그리고 두 가지 구동 모드

DB-native branching

  • 테이블 스키마·데이터 전체를 branch/merge
  • feature 브랜치에서 이슈 그래프를 마음껏 실험 → 본 main에 영향 0
  • dolt log / dolt diff로 데이터 변경을 코드처럼 리뷰
  • dolt push / dolt pull로 원격 DB와 동기화 (beads는 bd dolt push)

두 가지 모드

모드Writer언제
embedded1단일 에이전트·단일 CLI. 기본값, 가볍다.
serverN다중 워크트리 / 다중 에이전트 동시 쓰기. dolt sql-server.

SEOCHO .beads/는 on-demand로 sql-server를 띄운다. 워크트리 여러 개에서 동시에 bd가 돌 때 자동 승격.

03

git worktree

한 레포 · 여러 브랜치 · 여러 디렉터리

Concept

git worktree — 브랜치 전환의 블로킹을 제거

보통 워크플로는 브랜치 전환이 디렉터리 내용을 덮어쓰는 방식이다. 동시에 두 브랜치 작업 = 불가능.

repo 하나 .git/
=>
wt-runtime/
feat/runtime
wt-query/
fix/query-edge
wt-docs/
docs/readme

명령

# 새 워크트리를 브랜치로
git worktree add ../wt-runtime feat/runtime-readiness

# 등록된 워크트리 목록
git worktree list

# 정리
git worktree remove ../wt-runtime

얻는 것

  • 디렉터리 격리 — 한 쪽에서 pytest 돌려도 다른 쪽 파일 안 흔들림
  • 에이전트 한 명당 한 워크트리 = 병렬 실행 기본 단위
  • git stash 안 쓰고 컨텍스트 전환
How It Works

worktree는 어떻게 격리를 만드는가

"디렉터리 여러 개 = 저장소 여러 개"가 아니다. 하나의 object store(.git/objects)를 공유하고, 작업 상태만 분리한다.

원 워크트리 (메인)

seocho/
├── .git/
│   ├── objects/      ← 공유됨
│   ├── refs/         ← 공유됨
│   ├── HEAD          ← 자기 전용
│   ├── index         ← 자기 전용
│   └── worktrees/
│       └── wt-runtime/
│           ├── HEAD
│           └── index
└── (working files on HEAD)

linked 워크트리

wt-runtime/
├── .git              ← 파일 1개 (포인터)
│   "gitdir: /seocho/.git/
│    worktrees/wt-runtime"
└── (working files on
   feat/runtime-readiness)

.git이 디렉터리가 아닌 작은 텍스트 파일. 메인 레포의 하위 상태를 가리킴.

공유되는 것

objects/ (모든 커밋·트리·블롭), refs/ (브랜치·태그), config

분리되는 것

각 워크트리의 HEAD, index(staging), working tree 파일, hooks(선택)

불변

한 브랜치를 동시에 두 워크트리에 체크아웃 불가 (git이 거부). 안전장치.

04

SEOCHO에서의 활용

세 도구를 우리 워크플로에 어떻게 얹었나

Usage ①

이슈 정책 — bd 위에 얹은 라벨 축과 스크립트

active 이슈에 강제되는 7축 라벨

sev-*impact-*urgency-* sprint-*roadmap-* area-*kind-*

한 이슈를 보면 "얼마나 급한지 · 어디 영역인지 · 어느 스프린트/로드맵인지"가 한눈에 읽힌다. 누락 이슈는 린트가 실패시킨다.

계약서: docs/ISSUE_TASK_SYSTEM.md

운영 스크립트

scripts/pm/new-issue.sh       # 표준 이슈 생성
scripts/pm/new-task.sh        # 표준 task 생성
scripts/pm/sprint-board.sh \
  --sprint 2026-S03           # 보드 뷰
scripts/pm/lint-items.sh \
  --sprint 2026-S03           # 라벨·관계 검증
scripts/pm/bd-recover.sh      # 손상 DB 복구
CLAUDE.md §4.1~§4.3: 에이전트 전체 워크플로가 bd 중심으로 돈다. bd ready로 시작해서 bd close로 끝난다.
Usage ②

워크트리 + Dolt — 병렬 슬라이스의 기술 받침

슬라이스 하나 = 워크트리 하나

# 새 슬라이스 시작
git worktree add \
  ../wt-runtime-readiness \
  feat/runtime-readiness

# 워크트리 안에서는 --sandbox 권장
cd ../wt-runtime-readiness
bd --sandbox list
bd --sandbox update bd-42 \
  --status in_progress

--sandbox는 auto-sync 부수효과가 다른 브랜치로 새지 않게 한다. (CLAUDE.md §4.3)

Dolt가 뒤에서 하는 일

  • 첫 bd 호출 시 embedded 모드로 붙음
  • 두 번째 워크트리에서 동시에 쓰면 자동으로 sql-server 승격
  • 각 bd update가 Dolt 커밋 1건 ← 감사 로그
  • 셀 단위 merge 덕분에 두 워크트리가 같은 이슈를 고쳐도 안전

서버 상태: bd dolt status로 확인. 문제 생기면 진단(kill -QUIT) 먼저, 재시작은 나중.

05

Best Use Cases

세 도구가 각자 빛나는 순간 3가지

01
장기 에픽의 의존성 기반 진행

시나리오: "runtime contract stability" 같은 몇 주짜리 에픽. 하위 task가 수십 개, 순서도 있고, 작업 중 새 하위 task가 끊임없이 발견됨.

step 1

에픽 생성 bd create "..." -t epic -p 1. 하위 task들을 --deps blocks:<epic>로 연결.

step 2

에이전트가 세션 시작마다 bd ready → 블로커 없는 task만 노출. "어디부터 하지" 고민 제거.

step 3

작업 중 엣지케이스 발견 → bd create --deps discovered-from:<parent>. 부모와 링크된 채 그래프에 남는다.

step 4

6주 뒤 에픽 회고: bd dep tree <epic> + bd dolt log로 왜 이 task들이 생겼는지 전부 추적 가능.

왜 이게 베스트인가

  • 세션 경계를 넘어 기억이 유지됨 — 에이전트가 내일 붙어도 오늘의 맥락이 그래프에 있다
  • 자동 스케줄링 — bd ready가 다음 할 일을 결정해주니 사람이 중재 안 해도 됨
  • 감사 가능 — Dolt 커밋 로그가 "왜 이렇게 결정됐는지"를 그대로 보관
일반 이슈 트래커로 이걸 하면 두 달 뒤 "이 task가 왜 있더라?" 답이 안 나온다.
02
병렬 슬라이스 — 충돌 없는 다중 에이전트 개발

시나리오: 에이전트 A가 sdk-facade, B가 query-canonical, C가 runtime-shell을 같은 오후에 개발 중.

# 각 에이전트가 자기 워크트리 만든다
git worktree add ../wt-sdk    feat/sdk-split
git worktree add ../wt-query  fix/query-rerank
git worktree add ../wt-rt     feat/runtime-ready

# 각자 독립 bd claim
# A
cd ../wt-sdk && bd --sandbox update bd-a1 --claim
# B
cd ../wt-query && bd --sandbox update bd-b2 --claim
# C
cd ../wt-rt && bd --sandbox update bd-c3 --claim

# 각자 테스트 돌린다 — 파일 안 섞임
pytest tests/ -k sdk      # A
pytest tests/ -k query    # B
pytest tests/ -k runtime  # C

세 도구가 각자 맡는 역할

worktree

파일 시스템 격리 — A의 편집이 B의 pytest에 안 보임. 세 브랜치가 동시에 살아있음.

beads

세 에이전트가 각자 다른 bd-*를 claim. 의존 그래프는 공유되지만 작업 단위는 독립.

Dolt

sql-server 모드로 승격돼 세 워크트리의 bd 쓰기를 동시에 받는다. --sandbox로 auto-sync 부수효과 차단.

결과 리베이스 지옥 없음, 머지 충돌 최소, 리뷰 단위 명확.

03
스프린트 린트 — 셀 단위 머지로 살아남는 동시 업데이트

시나리오: 스프린트 마감 날, 자동화 여러 개가 같은 이슈를 동시에 업데이트한다.

동시 쓰기 3갈래

  • lint-items — 라벨 누락 보정: bd-42.sprint = 2026-S03
  • sprint-board — 상태 전이: bd-42.status = in_progress
  • 에이전트 — 코멘트 추가: bd-42.comments += "작업 시작"

셋이 1초 안에 같은 행을 건드린다.

일반 DB였다면

T1 write sprint
T2 read row → write status
  ← T1의 sprint 값 덮어쓰기
T3 write comments
  ← T2의 status 덮어쓰기
결과: comments만 살아남음

린트가 지워지고, 상태가 지워진다. 복구 스크립트 필요.

Dolt cell-merge로 실제 일어나는 일

세 쓰기가 각각 다른 컬럼을 건드리므로 전부 살아남는다. 동일 컬럼 충돌만 수동 resolve. bd dolt log에 세 커밋이 그대로 남아 누가 뭘 바꿨는지 감사 가능.

실전 체감 스프린트 자동화·에이전트·사람이 부딪혀도 데이터 유실 0. 이게 평범한 이슈 트래커였다면 매주 한 번은 복구 작업 필요.

Takeaway

세 도구, 세 레이어

레이어도구책임우리가 얻는 것
의미론beads 무엇이 작업인가, 무엇이 뭘 막는가 세션 경계를 넘는 메모리, 자동 스케줄링
데이터Dolt 다중 writer 하에서 데이터 무결성 셀 단위 머지, 커밋 단위 감사
물리git worktree 파일 시스템·브랜치 격리 병렬 실행, 충돌 제로 체크아웃
세 개가 서로의 약점을 덮는다. beads는 Dolt 없이는 동시 쓰기에 약하고, 워크트리 없이는 병렬 실행이 막힌다. 묶어야 AI 에이전트 규모를 감당한다.
