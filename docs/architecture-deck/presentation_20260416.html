01

What is SEOCHO?

Unstructured Data to Structured Knowledge Graphs

OVERVIEW

Vision & Stack Baseline

SEOCHO transforms unstructured data into ontology-governed Knowledge Graphs, then uses a dynamic Agent Pool with a Parallel Debate pattern to answer complex queries across multiple graph targets.
Agent Runtime OpenAI Agents SDK
Graph Backend DozerDB (Neo4j protocol compatible)
Trace / Evaluation vendor-neutral (none | console | jsonl | opik)
Tenancy Mode single-tenant MVP with workspace_id
Ontology Engine Owlready2 (offline governance only)
SDK Package pip install seocho

Package Structure

4-Layer Module Architecture

PACKAGES

Module Architecture

PACKAGES

Package Roles

seocho/ — Canonical SDK Engine

  • ontology*.py — ontology facade, context, serialization, artifacts, governance
  • query/ — intent, strategy, cypher builder, semantic agents, answer generation
  • agent/ — factory, context, contracts
  • index/ — pipeline, linker, extraction engine
  • store/ — graph, vector, LLM adapters
  • rules.py — SHACL-like inference & validation
  • client.py — public SDK facade
  • session.py — context persistence & caching

runtime/ — Deployment Shell

  • agent_server.py — FastAPI transport
  • server_runtime.py — service composition
  • memory_service.py — memory-first facade
  • policy.py — authorization
  • agent_readiness.py — ready / degraded / blocked

extraction/ — Compatibility Layer

Shims and aliases that delegate to seocho/* and runtime/*. Shrinking toward extraction-only concerns.

03

Control Plane vs Data Plane

Separation of Concerns

PLANES

Two-Plane Architecture

Control Plane

Agent orchestration, routing policy, runtime authorization, quality gates, ADR governance, ontology vocabulary lifecycle

Data Plane

Data ingestion (CSV/JSON/API), LLM 3-pass extraction, SHACL-like rule inference, graph load & Cypher query on DozerDB

04

End-to-End Data Flow

From Raw Data to Agent Answers

DATA FLOW

Ingestion to Query

DATA FLOW

Pipeline Steps Detail

1. Ontology Prompt Bridge

Ontology YAML/JSON-LD definitions are injected into LLM prompts as extraction context. Node types, relationship types, and property constraints guide the extraction.

NodeDef RelDef Jinja2

2. Extraction & Linking

LLM-based entity extraction followed by entity resolution. Embedding cosine similarity deduplicates entities across documents.

LLM 3-pass cosine dedup

3. Rule Constraints

SHACL-like rule inference annotates entities with required, datatype, enum, and range constraints before graph load.

SHACL Cypher constraints

4. Graph Provisioning

DatabaseManager creates databases in DozerDB, applies schema constraints, and loads MERGE operations via GraphLoader.

5. Agent Factory

Dynamically provisions one agent per graph target. Each agent's query tool is closure-bound to its specific DB context.

6. Query Execution

Three modes available: Legacy Router, Parallel Debate, or Semantic Graph QA. Details in the next section.

3 Execution Modes

Router / Debate / Semantic QA

EXECUTION

Three Query Execution Modes

LEGACY

1. Router Mode

POST /run_agent

Static 7-agent pipeline. Router delegates to exactly 1 specialist agent. Sequential hand-off chain.

GraphAgent VectorAgent WebAgent
PRIMARY

2. Parallel Debate

POST /run_debate

All DB-bound agents execute concurrently via asyncio.gather(). Supervisor synthesizes from Shared Memory. Fault-isolated.

fan-out shared memory synthesis
SEMANTIC

3. Semantic Graph QA

POST /run_agent_semantic

4-agent model with entity disambiguation. Fulltext lookup + semantic dedup before LPG/RDF routing.

entity resolution LPG/RDF split
DEBATE

Parallel Debate Pattern

Fan-Out

DebateOrchestrator spawns one agent per graph target. All agents execute concurrently. Each stores its isolated conclusion in request-scoped SharedMemory.

Synthesis

Supervisor Agent aggregates all results. If any agent fails, Supervisor synthesizes from remaining successful answers. Runtime payload includes agent_statuses and degraded flag.

SEMANTIC QA

Semantic Graph QA Flow

Semantic Layer

  • Question entity extraction
  • Fulltext index candidate search
  • Contains-based fallback search
  • Ontology-hint alias/label boost
  • Lightweight semantic dedup

Routing Policy

  • RDF hints (owl, shacl, sparql) -> RDFAgent
  • Default -> LPGAgent
  • Mixed hints -> hybrid path (both)
  • Frontend can pin entity_overrides

Ontology Module Design

Split Architecture & Boundary Rules

ONTOLOGY

Ontology Module Boundaries

Critical boundary: Owlready2 heavy reasoning is forbidden in the request hot path. Only allowed in the offline governance path (ontology_governance.py).
ONTOLOGY

Module Responsibilities

ontology.py — Public API Surface

Ontology, NodeDef, RelDef, P

Schema validation, SHACL derivation, prompt-facing API entrypoints

ontology_context.py — Runtime Hot Path

Context hash, SKOS glossary, in-process cache, query guardrails, graph-write metadata

ontology_artifacts.py — Runtime Artifacts

Typed artifact promotion, approved artifacts, semantic prompt context, vocabulary shaping

ontology_serialization.py — Persistence

Canonical JSON-LD load/save helpers. No runtime governance side effects.

ontology_governance.py — Offline Only

Offline check, diff, export, OWL inspection via Owlready2. Never in request hot path.

owlready2 offline only
# CLI surface for offline governance
seocho ontology check --schema schema.jsonld
seocho ontology diff --left old.jsonld --right new.jsonld
seocho ontology export --format shacl --output shacl.json

GUARDRAILS

Key Design Decisions

workspace_id Everywhere

All runtime write/compute APIs must include workspace_id. Single-tenant MVP but wired for multi-tenant.

DozerDB Safety

  • elementId() over deprecated id()
  • Dynamic labels validated before Cypher interpolation
  • Query tools read-safe by default

Owlready2 Boundary

Forbidden in request hot path. Only allowed in offline governance flows.

Agent SDK Isolation

All Agent SDK execution routed through agents_runtime.py. Never call Runner.run directly in feature modules.

Runtime Package Migration

extraction/ is shrinking. runtime/ is the target deployment shell. See docs/RUNTIME_PACKAGE_MIGRATION.md.

Trace Vendor Neutrality

JSONL as portable artifact. Opik as optional team exporter. Backend selectable via SEOCHO_TRACE_BACKEND env var.
