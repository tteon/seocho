Chapter 1 — Knowledge Graph Indexing

Source · Chunk · Entity 3-layer + Ontology-aware Extraction
seocho · ontology lab · 2026-S03

왜 이 챕터부터인가?

  • 그래프 품질·라우팅·debate 모두 인덱싱의 결과물 위에 얹힌다.
  • 여기서 만든 (:Source)-(:Chunk)-(:Entity) 구조와 community_id가 Ch 2~5의 입력.
  • 4-provider 비교는 인덱싱 단계가 가장 명확 — 같은 청크에 대한 추출 결과 직접 비교.
이 노트북이 통과하지 않으면 다음 4개 챕터가 모두 막힌다.

1.1 Source → Chunk → Entity (3-layer)

분리하지 않으면

  • 출처 추적 불가
  • 청크 단위 재계산 불가
  • 동일 엔터티 중복 노드 폭증
  • cascade 정책 모호

분리하면

  • (:Source)에 메타데이터 응집
  • (:Chunk) 부분 재인덱싱
  • (:Entity) dedup + 다대다 MENTIONS
  • DETACH DELETE 명확한 lineage
RDF로 같은 표현을 하려면 reification이 필요. LPG는 edge property로 끝.

Cypher: 3-layer 카운트

MATCH (src:Source)
OPTIONAL MATCH (src)-[:HAS_CHUNK]->(c:Chunk)
OPTIONAL MATCH (c)-[m:MENTIONS]->(e)
RETURN count(DISTINCT src) AS sources,
       count(DISTINCT c)   AS chunks,
       count(DISTINCT e)   AS entities,
       count(m)            AS mentions
회사명이 청크 5개에 등장 → Entity 1개 / MENTIONS 5개

1.2 Ontology Slice — 왜 발췌하는가

  • 전체 ontology를 prompt에 부으면 token + attention 낭비.
  • intent별로 관련 class/property만 선별 → 정확도 ↑, 비용 ↓.
  • Ch 3(text2cypher)에서 다시 등장 — query intent별 slice.
from seocho.ontology_slice import slice_ontology
sliced = slice_ontology(ontology, intent='risk', max_classes=8)

1.2 Extraction Prompt — 3 블록

  • Block 1: ontology slice (허용 class/property만)
  • Block 2: 강제 JSON schema {entities: [{class, name, evidence_span}]}
  • Block 3: evidence_span은 원문의 verbatim substring
evidence_span 강제는 hallucination을 가장 효과적으로 차단한다.

1.3 LPG Metadata 우위

MATCH (c:Chunk)-[r:MENTIONS]->(e)
WHERE r.confidence > 0.8
  AND r.extracted_by STARTS WITH 'gpt-4o'
RETURN e.name, r.confidence, r.extracted_at
edge property — RDF는 reification 없이는 못함.

1.4 Community Detection (Louvain)

  • Entity 그래프 위에 community 자동 분해.
  • 결과를 community_id property로 노드에 write-back.
  • Ch 4 라우팅에서 "이 질문은 어느 community?" 판단의 근거.
CALL gds.graph.project.cypher('ch01-finder',
  'MATCH (e) WHERE NOT e:Source AND NOT e:Chunk RETURN id(e) AS id',
  'MATCH (e1)<-[:MENTIONS]-(c:Chunk)-[:MENTIONS]->(e2)
   WHERE id(e1)<id(e2) RETURN id(e1) AS source, id(e2) AS target')
CALL gds.louvain.write('ch01-finder', {writeProperty: 'community_id'})
  YIELD communityCount, modularity
