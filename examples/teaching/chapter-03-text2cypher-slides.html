3.1 3-블록 Prompt

  • Block 1: ontology slice (관련 class/property만)
  • Block 2: few-shot 예제 3개 (easy / medium / hard)
  • Block 3: 출력 제약 — read-only, elementId, LIMIT 강제
3-블록 구조는 token 효율과 정확도를 동시에 끌어올린다.

Block 3 — Output Constraints

- Read-only: NO CREATE/MERGE/DELETE/SET
- Use elementId() instead of deprecated id()
- MANDATORY: every query must end with LIMIT (default 25)
- Output a single fenced cypher block, nothing else
LLM이 unbounded path (`*..`)를 생성하면 graph OOM. LIMIT 강제는 비용 안전장치.

3.4 TTL Korean Labels — 효과

fibo-be:Company
    rdfs:label "Company"@en ,
               "회사"@ko ,
               "기업"@ko ;
    skos:altLabel "Corporation"@en ,
                  "법인"@ko ;
    rdfs:comment "법인격을 갖는 사업체..."@ko .
ontology governance plane에서 label/synonym 풍부도가 한국어 질의 정확도를 결정한다.
TTL 수정은 data plane 인덱스에 영향 없음 (governance vs data 분리).
