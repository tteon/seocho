The dataset — FinDER

What it is

  • 5,703 expert-annotated query–evidence–answer triplets
  • Source: real 10-K filings from 490 S&P 500 companies
  • Scraped from EDGAR; annotated by an investment-bank analyst + a CPA
  • Available at huggingface.co/datasets/Linq-AI-Research/FinDER

Why it is hard

  • Queries are brief and ambiguous — real search behavior, not textbook QA
  • 43.45% of queries use 3–4 jargon terms; 46.41% use 5+
  • Acronyms collapse meaning: "MS trading revenue CAGR"
  • 49.83% of quantitative queries require compositional reasoning

Source: Choi et al., FinDER (ICLR '25 Advances in Financial AI). PDF in this folder.

FinDER's headline result — the context ceiling

From Table 8 of the paper: response correctness on the same models, same questions, only context provision changes:

SettingClaude-3.7-SonnetGPT-o1DeepSeek-R1-DistillQwen-QWQ
Without context9.3710.149.889.13
Top-10 retrieved (E5-Mistral)33.8932.9628.7929.41
Perfect (gold) context66.4868.1359.6961.03
The lesson: the dominant lever in financial QA isn't the LLM, it's the quality of the retrieved context. Closed-book LLMs answer ~10% of expert-grade queries. Top-10 dense retrieval recovers ~30%. Gold context recovers ~65%.

This is exactly the gap an ontology-grounded retrieval pipeline tries to close.

Why ontology — the disambiguation gap

The FinDER paper opens with this canonical example:

User query: "MS trading revenue CAGR"
Inaccurate retriever returns: Microsoft's talent acquisition page → LLM says "Sorry, no trading revenue for Microsoft."
Accurate retriever returns: Morgan Stanley's investment-banking statements → LLM computes 9% CAGR correctly.

The fix is not a bigger LLM. The fix is schema — a structure that distinguishes Corporation { ticker: "MS", name: "Morgan Stanley" } from Corporation { ticker: "MSFT", name: "Microsoft" } at retrieval time.

FIBO already encodes that structure for the financial domain.

FIBO — what it is & what we use

FIBO at a glance

  • Financial Industry Business Ontology — open standard from the EDM Council
  • OWL 2 DL, MIT-licensed, polyhierarchical
  • Built by SMEs across banks, regulators, vendors
  • Authoritative URL: spec.edmcouncil.org/fibo/

What this repo ships

  • examples/datasets/fibo_be_minimal.ttl
  • 5 classes: Party · Person · LegalPerson · FormalOrganization · Corporation
  • 4 datatype properties: hasName · hasTitle · hasIdentifier · hasLegalName
  • 2 object properties: isOfficerOf · hasShareholder
  • Every class carries skos:exactMatch back to the canonical FIBO IRI

Hand-curated for tutorials. The pattern transfers to any OWL/Turtle source — including the full FIBO release.

FIBO's quality discipline (read before you extend it)

From the FIBO best-practices guide (companion file in this folder). The principles are not optional:

PrincipleWhat it means in seocho
Use-case driven — every element ties to a competency questionEach NodeDef/RelDef you add should answer at least one FinDER-style query
ISO 704 definitions — partial sentence, genus/differentia, non-circularGoes into NodeDef(description=...) — it ends up in the LLM extraction prompt verbatim
Naming — UpperCamelCase classes, lowerCamelCase verb properties, no abbreviationsseocho passes names through unchanged into Cypher labels — keep them clean from day one
Logical consistency — provably consistent under multiple OWL 2 reasonersOwlready2 reasoning is the offline gate (CLAUDE.md §6.3); never run heavy reasoning in the request hot path
Unique labels — no two classes/properties share a label across namespacesHash collision in compile_ontology_context() if you violate this
Hygiene tests (21 of them) — labels present, no circular defs, no min 1Mirrored by seocho/tests/test_ontology_*.py — your extension PR runs these

The hands-on workflow — bundled sample

Loop the FinDER-shaped sample through the full seocho stack. Run from examples/:

import json
from pathlib import Path
from seocho import Seocho, AGENT_PRESETS

# 1. Load FIBO subset (Tutorial 3 §2 covered this loader)
from tutorial_03_helpers import load_fibo_ontology
ontology = load_fibo_ontology(Path('datasets/fibo_be_minimal.ttl'))

# 2. Stand up a workspace-stamped client
s = Seocho.local(
    ontology,
    llm='openai/gpt-4o-mini',
    agent_config=AGENT_PRESETS['agent'],
    user_id='analyst@team',
    workspace_id='finder-eval',
)

# 3. Ingest the FinDER-shaped sample (10 cases, same 8 categories)
sample = json.loads(Path('datasets/tutorial_filings_sample.json').read_text())
with s.session('finder-smoke') as sess:
    for case in sample:
        sess.add(case['text'])

# 4. Ask each query and compare to expected_answer
    for case in sample:
        got = sess.ask(case['question'])
        print(case['id'], case['category'], '→', got[:120])

Every span emitted carries workspace_id="finder-eval", user_id="analyst@team", and the context_hash of fibo_be_minimal. That is your audit log.

Scaling up — the real FinDER corpus

The bundled sample is for smoke. For benchmark-grade evaluation:

from datasets import load_dataset

finder = load_dataset('Linq-AI-Research/FinDER', split='test')
# columns: query · evidence · answer · category · reasoning_type · company

with s.session('finder-full') as sess:
    # ingest ground-truth evidence (gold-context setting from Table 8)
    for row in finder:
        sess.add(row['evidence'], metadata={'company': row['company']})

    correct, faithful = 0, 0
    for row in finder:
        got = sess.ask(row['query'])
        # use RAGAS metrics (Correctness + Faithfulness) to score
        correct  += score_correctness(got, row['answer'])
        faithful += score_faithfulness(got, row['evidence'])
Boundary: never publish results of the bundled tutorial sample as benchmark evidence. Use the HuggingFace dataset, run RAGAS, report the same Correctness/Faithfulness pair as the FinDER paper (Table 7).

What seocho gives you that the FinDER baseline doesn't

The FinDER paper's baseline pipeline is retriever (E5-Mistral) → reranker → LLM. seocho adds three things that move from "good evaluation" to "production audit":

NeedFinDER baselineseocho
Why this answer?Top-k passage IDscontext_hash + ontology-typed nodes/edges + elementId(...) in Cypher
Whose run was that?Notebook cell metadatauser_id + workspace_id on every span (JSONL or Opik)
What does v2 of the schema break?Manual diffontology_v1.migration_plan(ontology_v2) → inspectable Cypher list
Did the retriever silently fall back?—span.metadata.degraded + fallback_from + fallback_reason
Are inferred rules pinned to the schema?n/ainfer_rules_from_graph() stamps ontology_identity_hash

Suggested teaching plan (90 min)

  1. 10 min — Recap the three-layer stack from the seocho tutorials.
  2. 15 min — Walk the FinDER paper: motivation, 8 categories, MS-vs-MSFT, the context-ceiling table.
  3. 10 min — Walk the FIBO best-practices guide (use-case discipline, ISO 704, hygiene tests).
  4. 20 min — Hands-on: load FIBO subset, ingest tutorial_filings_sample.json, ask each query, inspect spans.
  5. 15 min — Extend the ontology with one new class motivated by an unmet competency question (e.g. BusinessSegment for case_002); run migration_plan.
  6. 10 min — Switch to the HuggingFace FinDER split, score Correctness + Faithfulness, compare to Table 7.
  7. 10 min — Inspect Opik traces; filter by workspace_id; show drift detection on the v1→v2 hash change.

The contract, restated for this dataset

FieldFor FinDER specifically
context_hashOne hash per FIBO subset version → re-running with FIBO-2025-Q3 produces a different hash, every span tells you which
workspace_idUse one workspace per evaluation split (finder-test, finder-train) so you never cross-contaminate retrieval indexes
user_idOne per analyst running the eval — Opik filter by metadata.user_id shows exactly that person's runs
category (FinDER native)Pass into session.add(metadata={'category': row['category']}); it shows up on the persisted node
degraded / fallback_fromIf the agent path raised, the pipeline path took over — your eval row is not invalid, but it should be flagged separately

If FIBO ships a new release and you re-run, every old artifact stays tagged with the old hash. You can prove which schema produced which answer.
