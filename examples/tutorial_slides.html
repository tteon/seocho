The instruction-design lever

Same model, same backend. Three system prompts produce three different output shapes:

# Pattern 1 — role + scope
system='You extract company names ONLY. Reply as a comma-separated list.'

# Pattern 2 — format constraint with JSON mode
system='Return JSON: {"pairs": [{"person": str, "organization": str}]}'
backend.complete(system=..., user=..., response_format={'type': 'json_object'})

# Pattern 3 — few-shot in the system prompt
system='''Extract executive titles. Return JSON: {name, title, organization}.

Example:
  Input: "Sundar Pichai is the CEO of Google."
  Output: {"name": "Sundar Pichai", "title": "CEO", "organization": "Google"}
'''

The system prompt is the largest quality lever you have. The next slide is why that matters.

...but hand-written prompts don't scale

Each new entity type, each new constraint, each new SHACL rule = another paragraph in the system prompt. Drift is invisible. Output schema is a string convention, not a contract.

An ontology is a structured way to generate the relevant part of that prompt automatically — and to validate the output against it.

ext_ctx = ontology.to_extraction_context()
system = (
    f'You extract entities for the {ext_ctx["ontology_name"]} graph.\n'
    f'Entity types:\n{ext_ctx["entity_types"]}\n'
    f'Relationship types:\n{ext_ctx["relationship_types"]}\n'
    'Return JSON: {"nodes": [...], "relationships": [...]}'
)

Tutorial 3 — what is FIBO?

FIBO in one paragraph

The Financial Industry Business Ontology is the EDM Council's open-standard ontology for finance. MIT-licensed. Released by domain experts (banks, regulators).

Authoritative URL: spec.edmcouncil.org/fibo/

What we use today

A teaching subset: fibo_be_minimal.ttl

  • 5 classes: Party · Person · LegalPerson · FormalOrganization · Corporation
  • 4 datatype properties: hasName · hasTitle · hasIdentifier · hasLegalName
  • 2 object properties: isOfficerOf · hasShareholder

All five classes carry skos:exactMatch back to canonical FIBO IRIs.

Hand-curated for tutorials — not a FIBO release. The pattern transfers to any OWL ontology.

From .ttl to seocho.Ontology

owlready2 doesn't parse Turtle natively — it dispatches to NTriples. We pre-convert via rdflib:

def load_fibo_ontology(ttl_path, *, ontology_name='fibo_be_minimal'):
    import owlready2 as o2, rdflib, tempfile, os
    g = rdflib.Graph()
    g.parse(str(ttl_path), format='turtle')

    # owlready2 → RDF/XML hand-off via tempfile
    with tempfile.NamedTemporaryFile(suffix='.owl', delete=False) as tmp:
        owl_path = tmp.name
    try:
        g.serialize(destination=owl_path, format='xml')
        onto = o2.get_ontology(f'file://{owl_path}').load()
    finally:
        os.unlink(owl_path)

    # walk classes / data_properties / object_properties → seocho.Ontology
    return Ontology(name=ontology_name, nodes={...}, relationships={...})

Output: Ontology(nodes=5, relationships=2, validate=OK).

The "FIBO modifying" approach

FIBO doesn't model product offerings (it stops at the financial-instrument boundary). To capture "Apple announced the iPhone", we extend FIBO with a non-FIBO Product node — without editing the source TTL.

extension = Ontology(
    name='product_extension', version='1.1.0',
    description='Non-FIBO product extension layered on top of FIBO BE.',
    nodes={
        'Product': NodeDef(
            description='A named product offered by a corporation.',
            properties={'hasName': P(str, unique=True),
                        'hasCategory': P(str)},
        ),
    },
    relationships={
        'MAKES': RelDef(source='Corporation', target='Product',
                        cardinality='ONE_TO_MANY'),
    },
)

ontology_v2 = ontology_v1.merge(extension, strategy='union',
                                 name='fibo_be_minimal+product')
ontology_v2.version = '2.0.0'

Why merge instead of edit? Your derivation stays transparent. migration_plan works out of the box. Lineage back to canonical FIBO is preserved.

migration_plan: what changed?

plan = ontology_v1.migration_plan(ontology_v2)

A migration is a plan, not a side effect. The Cypher statements are inspectable, runnable on demand, and can be replayed.

Why ontology compile_context matters

from seocho import compile_ontology_context

compiled_v1 = compile_ontology_context(ontology_v1, workspace_id='tutorial')
compiled_v2 = compile_ontology_context(ontology_v2, workspace_id='tutorial')

print(compiled_v1.descriptor.context_hash)   # → '7a4b...'
print(compiled_v2.descriptor.context_hash)   # → 'c3f0...'  (changed!)
Every IndexingResult stamps the context_hash on its output.
Every trace span carries context_hash in metadata.
Every RuleSet from infer_rules_from_graph carries an ontology_identity_hash.

Promotion gates compare hashes: match → apply / drift → block / unknown → require review. The audit trail is built into the data.

Identity: user_id + workspace_id

Multi-user projects need to attribute traces. seocho 0.3.1 wires both fields through every emitted span.

s = Seocho.local(
    ontology=ontology_v2,
    llm='openai/gpt-4o-mini',
    user_id='openai-yitae',     # ← attribution
    workspace_id='acme',        # ← tenancy
)
with s.session('demo') as sess:
    sess.add('Tim Cook leads Apple.')
# Span metadata now carries:
#   user_id      = 'openai-yitae'
#   workspace_id = 'acme'
#   context_hash = 'c3f0...'

Filter Opik for metadata.user_id = 'openai-yitae' and you see exactly that user's runs across the whole project.
